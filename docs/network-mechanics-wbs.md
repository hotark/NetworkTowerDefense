## MS0: レベリング調整ツール（バランスシミュレータ）

configの数値から待ち行列理論に基づいてρ・スループット・稼働率を計算し、
レベル組み合わせごとのボトルネックを可視化する開発用ツール。

### 概要
- `tools/balance-sim/index.html` — 単体HTMLツール（ゲーム非依存）
- config値を入力 → 理論値を即座に表示
- ゲーム実装前にバランスを机上検証できる

### 入力パラメータ
- 生成器: 生成間隔（λ = 1/interval）、レベル別
- エッジ: 容量（最大パケット保持数）、パケット通過速度（μ）、レベル別
- タワー: 弾薬消費率（μ_demand）、発射間隔、レベル別
- リピーター: 増幅数（出力パケット数）、レベル別
- 分配器: 分岐数（maxFanout）、レベル別

### 出力・表示
- **ρ（利用率）テーブル**: エッジLv × 生成器Lv の全組み合わせでρを計算、色分け表示
  - ρ < 0.5 緑（余裕）、0.5-0.8 黄（注意）、0.8-1.0 赤（輻輳）、>1.0 黒（破綻）
- **タワー稼働率テーブル**: タワーLv × 供給構成（エッジLv + リピーター有無）で稼働率表示
  - 稼働率 = min(1, λ_supply / μ_demand)
  - 100% = フル稼働、50% = 半分しか撃てない
- **ボトルネック判定**: 「タワーLv3 + エッジLv1 → 稼働率33%、エッジLv2推奨」のような提案
- **Little's Law検証**: L = λW で各エッジの平均キュー長を予測
- **構成シミュレータ**: ノード構成を選んで（生成器→リピーター→分配器→タワー×2 等）全体スループットを算出

### タスク
- [ ] config値の入力UI（スライダー or 数値入力、レベル別タブ）
- [ ] ρ計算エンジン（M/M/1/K モデル: 有限キュー長）
- [ ] エッジLv × 生成器Lv のρマトリクス表示
- [ ] タワーLv × 供給構成の稼働率マトリクス表示
- [ ] ボトルネック自動判定ロジック
- [ ] 構成シミュレータ（ノード接続をGUIで組む or プリセット選択）
- [ ] 結果のJSON/CSV出力（config.jsにそのまま使える形式）

### 完了条件
レベル値を変えると即座にρと稼働率が更新され、どの組み合わせが破綻するか一目で分かる。

## MS1: エッジ容量と輻輳

エッジに最大パケット保持数を持たせる。満杯なら入れない。それだけ。
待ち行列理論はMS0のツール側で扱い、ゲーム内ロジックはシンプルに保つ。

### ゲーム内ロジック（シンプル）
- エッジに `capacity`（最大保持数）を持つ
- パケット送出時: 現在のパケット数 < capacity → 通る、>= capacity → 入れない（待機 or 別経路）
- 輻輳判定: `現在数 / capacity` の比率だけ見る

### タスク
- [ ] エッジにcapacity属性を追加
- [ ] パケット送出時の容量チェック（満杯→ノードで待機）
- [ ] 輻輳エフェクト（パケット数/capacityが高い → エッジ色変化: 青→黄→赤）
- [ ] エッジレベルによるcapacity変化（Lv1: 3, Lv2: 5, Lv3: 8）
- [ ] エッジアップグレードUI（コスト消費で容量拡張）

### 完了条件
パケットが溜まったエッジが赤くなり、満杯だとパケットが入れなくなる。

## MS2: タワーレベルと弾薬消費スケーリング

タワー強化 → 弾薬消費増 → ネットワーク強化要求の連鎖を実装。

### 理論的基盤
- タワーLvが上がるとサービス要求率（弾薬消費μ_demand）が増加
- ネットワーク供給率λ_supply < μ_demand になるとタワーが弾切れ → 性能を発揮できない
- 供給を増やすにはλを上げる（リピーター）かμを上げる（エッジ強化）

### タスク
- [ ] タワーレベル別の弾薬消費率を定義（Lv1: 1/s, Lv2: 3/s, Lv3: 6/s, Lv4: 10/s）
- [ ] タワーアップグレード機能（コスト・所要時間）
- [ ] アップグレード中の動作停止
- [ ] 弾薬不足時の警告表示（弾切れ頻度の可視化）
- [ ] リピーター/分配器のレベルアップ機能
- [ ] コスト設計: ネットワーク系ノード高め、タワー低め
- [ ] バランス検証: 各Lvで必要なネットワーク構成を確認

### 完了条件
タワーをLv2にしたら弾切れが起き、エッジ強化 or リピーター追加で解消される体験。

## MS3: 冗長性とエッジ障害

エッジ破壊と代替経路による冗長性メカニクスを実装。

### 理論的基盤
- ネットワーク信頼性: 単一障害点(SPOF)の排除
- 経路冗長度: min-cut / max-flow（フォード・ファルカーソン的考え方）
- 可用性 = 1 - Π(1 - p_i)（並列冗長の信頼性）

### タスク
- [ ] エッジにHP属性を追加
- [ ] 敵の特殊攻撃: エッジにダメージを与える敵タイプ
- [ ] エッジ破壊時のパケット消失と経路再探索
- [ ] 破損エッジの修理機能（コスト + 時間）
- [ ] 冗長経路の自動フォールバック（DAGで代替経路があれば自動切替）
- [ ] 損傷エッジの可視化（ひび割れ、点滅）
- [ ] ノード破壊（HP 0 → 接続エッジ全滅）の実装
- [ ] バランス検証: 冗長構成のコスト対効果

### 完了条件
エッジが破壊されても冗長経路でパケットが流れ続け、修理で復旧できる。

## 実装順序の依存関係

```
MS1（エッジキュー）
  ↓
MS2（タワーLv + 弾薬消費）← MS1のキューがないと輻輳が体験できない
  ↓
MS3（冗長性）← MS1,2が動いている前提でエッジ障害を入れる
  ↓
MS4（アップグレード停止）← MS2のアップグレードがベース
  ↓
MS5（ビジュアル）← ロジックが固まってから演出を載せる
  ↓
MS6（バランス）← 全メカニクス統合後に調整
```
